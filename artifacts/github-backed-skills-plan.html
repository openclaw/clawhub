Internal explainer · current plan

GitHub-backed skills without mirroring

ClawHub remains the catalog and security gate. GitHub remains the source of truth. OpenClaw installs live from GitHub only when the current skill-folder hash matches the hash ClawHub scanned.

Core invariant: for GitHub-backed skills, install/update succeeds only when current GitHub skill-folder hash == last scanned skill-folder hash.

Decisions We Have Made

1

No mirror. ClawHub will not serve stored copies for GitHub-backed skills like NVIDIA skills.

2

Normal user UX. Users still run openclaw skills install nvidia/foo; OpenClaw learns from ClawHub that this is GitHub-backed.

3

Strict hash gate. If current GitHub content does not match the scanned content hash, install/update fails as verification pending.

4

Folder hash beats repo SHA. Repo commit can change without changing a skill. The security check should compare the skill-folder content hash.

5

No ref or manifest path in DB yet. V1 treats GitHub-backed skills as default-branch installs; future sync can add repo-level config if needed.

6

Sync postponed. The first work enables GitHub-backed install plumbing. Metadata sync and scan kickoff come later.

DB Schema Mental Model

Keep the existing hosted-artifact model. Add one small GitHub install-source object and one compact verification summary to skills. Do not add a separate revisions table for V1.

Catalog identity: slug, display name, owner, moderation, latest version pointer, tags, stats.

Current skillVersions row

skillVersions.sourceProvenance means “this stored artifact came from GitHub.” It does not mean “install live from GitHub.”

GitHub-Backed Install Flow

OpenClaw
1. User runs normal ClawHub install

openclaw skills install nvidia/foo. OpenClaw asks ClawHub for install instructions.

ClawHub
2. Detects GitHub-backed skill

Loads skills.installSource.kind = "github" plus repo/path and latest scanned content hash.

ClawHub
3. Live-checks GitHub

Fetches current skill-folder hash for repo default branch and path. The repo commit may differ; folder hash decides safety.

ClawHub
4. Compares hashes

If current hash equals last scanned hash, ClawHub returns a source descriptor. If not, install fails as verification pending.

OpenClaw
5. Installs pinned source

OpenClaw fetches from GitHub at the commit ClawHub resolved and installs the configured subdirectory.

Failure rule: if current.contentHash !== githubVerification.contentHash, do not install the old scanned content and do not install the new unscanned content. 

if skill.installSource.kind == "github":
  current = resolveGitHubFolder(repo, path) // default branch

  if current.status == "removed":
    fail("removed upstream")

  if current.contentHash != skill.githubVerification.contentHash:
    fail("verification pending")

  return {
    kind: "github",
    repo,
    path,
    commit: current.commit
  }

Direct Upload Stays The Same

Hosted ClawHub skill
  • Publisher uploads files to ClawHub.
  • ClawHub stores files as a skillVersions row.
  • ClawHub scans those exact stored files.
  • OpenClaw downloads those exact files from ClawHub.
Invariant

The thing scanned is the thing installed.

skillVersion.files
  -> ClawHub scan
  -> /api/v1/download
  -> OpenClaw install

Postponed On Purpose

Metadata sync Later

No cron, GitHub Action, GitHub App, or repo registration UI in this first chunk.

Manifest priority Later

Do not store manifestPath or configurable refs yet. Later code can try ClawHub schema first, then skills.sh.json.

Scan kickoff Later

How new GitHub content enters ClawHub scanning is separate. V1 only consumes existing scan fields.

This means early testing needs manually populated GitHub-backed skill fields, a staff-only mutation, or seeded data. The automated sync pipeline is deliberately out of scope.

FAQ

Why not a new scan snapshot table?

Because the first feature only needs the latest scanned content hash and status. Add history only when we need audit trails or large scan payloads.

Why not store githubRef?

For V1, GitHub-backed skills resolve the repo default branch by policy. Configurable refs belong to later source-sync configuration, not each skill row.

Why compare content hash instead of commit SHA?

A monorepo commit can change other skills. The safe check is whether the target skill folder bytes match what ClawHub scanned.

What if GitHub changed but the skill folder did not?

Allow install. The folder hash still matches the scan.

What if the skill folder changed?

Fail install/update with verification pending until the future sync/scan path updates githubVerification.contentHash.

What about NVIDIA OMS signatures?

For V1, signature verification is represented in ClawHub’s stored status. OpenClaw does not add a separate local OMS gate.

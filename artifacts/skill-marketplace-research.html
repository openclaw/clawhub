Research snapshot · 2026-06-01

Marketplace JSON is a catalog, not the skill payload.

The common pattern across Claude, Codex, Copilot, and Hermes is: publish skills as folders, then optionally wrap them in a marketplace/plugin manifest so an agent can discover, install, cache, and update them.

TL;DR

For ClawHub auto-sync from NVIDIA/skills, treat the repo's skills/ tree as the primary skill catalog. Treat .claude-plugin/marketplace.json and .agents/plugins/marketplace.json as secondary bundle metadata.

1. Checkout repo Use GitHub Actions against a pinned ref or the current main SHA.
2. Scan skill roots Default root: skills/. Curated plugin subset: plugins/nvidia-skills/skills/.
3. Dry run first clawhub sync --dry-run --json --owner nvidia --source-repo NVIDIA/skills
4. Publish changed skills Use the existing reusable skill-publish.yml workflow and preserve source provenance.
Why this matters: NVIDIA's marketplace files currently advertise a single nvidia-skills plugin bundle. The repo's root skills/ directory is the broad catalog. Syncing only the marketplace entry would miss most standalone skills.

Vocabulary

Skill A folder with SKILL.md and optional support files.

Plugin A bundle that can include skills, agents, hooks, MCP servers, assets, or app metadata.

Marketplace A JSON catalog that points at plugins or bundles.

Registry A service like ClawHub that stores, searches, audits, and installs published artifacts.

Format Comparison

Claude Code marketplace

Claude expects .claude-plugin/marketplace.json at the marketplace root. Each entry has a name and source; the source can be a relative path, GitHub repo, git URL/subdirectory, URL, or npm package. 

{
  "name": "company-tools",
  "owner": { "name": "DevTools Team" },
  "plugins": [
    {
      "name": "quality-review-plugin",
      "source": "./plugins/quality-review-plugin",
      "description": "Adds a quality-review skill"
    }
  ]
}

Codex marketplace

Codex uses .agents/plugins/marketplace.json. The OpenAI sample spec makes the marketplace more UI/policy-shaped: interface.displayName, ordered plugin entries, a local source.path, policy, and category. 

{
  "name": "openai-curated",
  "interface": { "displayName": "ChatGPT Official" },
  "plugins": [
    {
      "name": "linear",
      "source": { "source": "local", "path": "./plugins/linear" },
      "policy": { "installation": "AVAILABLE", "authentication": "ON_INSTALL" },
      "category": "Productivity"
    }
  ]
}

NVIDIA/skills today

NVIDIA ships both marketplace dialects and both plugin manifests. The marketplace points to ./plugins/nvidia-skills; the Claude plugin manifest lists skills: ["./skills/"], while the Codex manifest uses skills: "./skills/" plus richer UI metadata.

Root catalog

Repo-level skills/ contains the large skill catalog.

Curated plugin bundle

plugins/nvidia-skills/skills/ contains a smaller bundled subset for agent plugin delivery.

Hermes / NousResearch

Hermes documents several hub sources: official skills, skills.sh, well-known endpoints, direct GitHub skills/taps, ClawHub, and Claude marketplace-style repos. So yes, Nous/Hermes mentions this pattern, but it treats Claude marketplaces as one adapter among many. 

Hermes source shapes:
official | skills-sh | well-known | url | github | clawhub | claude-marketplace | lobehub | browse-sh

ClawHub sync

ClawHub's existing path is already skill-folder first: clawhub sync scans folders containing SKILL.md, hashes files, checks the registry, and publishes new or changed versions. The reusable workflow wraps that with source provenance. 

bun packages/clawhub/src/cli.ts sync \
  --all --json --no-clawdbot-roots \
  --owner nvidia \
  --source-repo NVIDIA/skills \
  --source-commit "$GITHUB_SHA" \
  --source-ref "$GITHUB_REF"

Decision Table

Question Use Reason
Do we want every NVIDIA standalone skill on ClawHub? skills/ This is the broad source tree and matches ClawHub's current skill publish model.
Do we want the curated NVIDIA plugin bundle? plugins/nvidia-skills/ This is the installable Claude/Codex bundle with its own manifests and assets.
Do we need to mirror Claude/Codex discovery? marketplace.json Use it as bundle inventory and presentation metadata, not as the canonical skill list.
Do we need automatic sync? GitHub Actions + skill-publish.yml ClawHub already records source repo, commit, ref, path, and JSON dry-run output.

Sharp Edge

Do not infer all skills from the marketplace file. In NVIDIA's repo, the marketplace advertises a plugin bundle. It does not enumerate the full root skills/ catalog one by one.
Do not publish from mutable HEAD without provenance. Always attach sourceRepo, sourceCommit, sourceRef, and derived sourcePath so ClawHub can prove exactly what was synced.

Proposed ClawHub Contract

Phase 1: skill sync

  • Add a caller workflow in NVIDIA/skills that invokes ClawHub's reusable skill-publish.yml.
  • Default root: skills for the broad catalog.
  • Use workflow_dispatch first: dry-run, publish-single, publish-catalog.
  • Only enable push-to-main automation after dry-run noise is low.

Phase 2: plugin/bundle awareness

  • Parse .claude-plugin/marketplace.json and .agents/plugins/marketplace.json as package metadata.
  • Publish the plugin bundle separately from standalone skills.
  • Surface bundle membership on ClawHub, but keep individual skill pages installable.
  • Track marketplace drift as an advisory signal, not as the only source of truth.

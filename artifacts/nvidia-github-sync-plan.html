GitHub-backed NVIDIA skills in ClawHub

ClawHub will index and verify NVIDIA skills from GitHub while keeping NVIDIA/skills as the canonical source of truth.

Goal: ClawHub becomes a trusted catalog and install gateway, not a second place where NVIDIA skills are independently republished.
The concern

A hosted copy can drift from the signed upstream

Current copy risk

If ClawHub stores and serves its own copy, "Official NVIDIA" can become ambiguous: is it NVIDIA's signed repo content, or a ClawHub artifact uploaded under NVIDIA's publisher?

Desired trust model

NVIDIA publishes once to GitHub. ClawHub resolves the live upstream state, verifies the exact revision, scans it, and only serves approved upstream content.

We are explicitly avoiding a mirror-by-default model for GitHub-synced NVIDIA skills.
Proposed V1 model

Source-backed catalog, proxied install

NVIDIA/skills

Canonical repo containing SKILL.md, companion files, skill-card.md, eval artifacts, and signature bundles.

>
ClawHub

Stores metadata, provenance, scan results, signature status, and the last verified commit. Live-checks upstream before install/update.

>
OpenClaw

Installs by slug through ClawHub, which resolves the current upstream ref and serves a verified pinned GitHub revision.

Sync contract

Publisher-triggered sync is the fast path

GitHub Action sends a pointer
{ "repo": "NVIDIA/skills", "ref": "refs/heads/main", "commit": "40-char-git-sha", "manifestPath": "skills.sh.json" }
Install/update still checks GitHub
  • No publisher-supplied artifact payload.
  • No uploaded zip treated as NVIDIA truth.
  • Every install live-resolves upstream, then serves an exact repo, path, and commit.
The Action is a pre-verification cache warmer. Correctness does not depend on it running successfully.
Verification and scanning

Only a verified commit becomes installable

Fetch the exact GitHub commit from the canonical repo.
Validate skill folder structure and parse SKILL.md metadata.
Verify skill.oms.sig where present against the fetched files.
Run ClawHub security scanning using the same policy as ordinary uploads.
Record repo, path, commit SHA, content hash, signature status, and scan result.
Install only the current upstream revision after it passes. If upstream changed or disappeared, fail visibly instead of serving stale content.
Versioning model

Commit SHA is the source identity

SHA Required source revision for install, update, and provenance.
hash Required canonical content hash for fetched skill files.
version Optional human label from SKILL.md or metadata.version.
latest Means current upstream revision after verification, not highest semver.
Version: 2.1.0 Source revision: abc123def456 If no upstream version exists: Source revision: abc123def456
Install and update UX

Keep the ClawHub slug workflow

User command stays simple
openclaw skills install nemoclaw-user-configure-security openclaw skills update nemoclaw-user-configure-security
Under the hood
  • ClawHub resolves slug to the current upstream GitHub state.
  • If current upstream is verified, install content is served from that pinned SHA.
  • If upstream changed or was removed, install blocks or fails visibly.
  • OpenClaw records installed source revision for future update checks.
V1 scope

Small, explicit, partner-safe

Included
  • GitHub Action-triggered sync.
  • Live upstream check on install/update.
  • skills.sh.json grouping support.
  • Source provenance and revision display.
  • Signature and scanner gate before installability.
Not included in V1
  • No automatic polling cadence.
  • No ClawHub-hosted mirrored artifact.
  • No GitHub App requirement yet.
  • No local OpenClaw identity redesign.
V2 path
  • GitHub App for tighter repo authorization.
  • Webhook-based reconcile.
  • Richer signature trust policies per publisher.
  • Partner dashboard for sync state.
Decision we are proposing

ClawHub as downstream verifier, not second publisher

NVIDIA remains the publisher of record in GitHub. ClawHub indexes, live-resolves, verifies, scans, and routes installs to exact upstream commits.

This addresses the core security concern: ClawHub cannot silently serve a different or stale "official NVIDIA" artifact than the current content NVIDIA signed and published in its canonical repository.

Open question for alignment: what NVIDIA certificate or trust root should ClawHub use when validating skill.oms.sig in production?
